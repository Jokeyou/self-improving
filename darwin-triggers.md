# Darwin Skill 自动触发规则

> 定义何时触发达尔文 skill 的优化流程。
> 设计时间：2026-04-19

---

## 触发机制

### T1 · 用户纠正触发（核心，最有价值）

**原理：** 用户纠正一次，说明 skill 实际输出与预期有偏差。积累同类型偏差 = skill 需要优化。

**规则：**
1. 每次用户纠正 Vex 的输出，在 `corrections.md` 中记录时，标注涉及哪个 skill
2. 每轮 heartbeat 检查 corrections.md 中近 7 天的新增纠正
3. 同一 skill 累计 **≥ 2 次同类型纠正** → 触发到达尔文优化流程

**同类型定义：**
- 同一维度（格式/内容/语气/专业性）
- 同一 skill 或同一领域

**触发后的动作：**
- 在 heartbeat 时告知用户："[skill-X] 近7天被纠正2次，是否启动优化？"
- 用户确认 → 启动达尔文单 skill 优化
- 用户拒绝 → 记录 "已忽略，继续观察"

---

### T2 · 使用热度触发

**原理：** 高频使用的 skill，如果长期没有 feedback，说明它正在"负重工作"但没有被优化。

**规则：**
1. 每周日 10:00 cron 触发，运行 `scripts/skill-usage-stats.py`
2. 扫描最近 7 天 session，统计 skill 关键词出现次数
3. 阈值：出现 ≥ 5 次 且 无 correction 记录 → 触发健康检查
4. 结果推送飞书："[skill-X] 本周被提到 N 次，是否做一次质量检查？"

**触发后的动作：**
- 告知用户："[skill-X] 本周被提到 N 次，是否做达尔文评估？"
- 用户确认 → 运行达尔文评估（只评不改，用户看完报告再决定是否优化）

**与 T1 的关系：**
- T1 看 correction 记录（质量信号）
- T2 看 session 统计（使用频率）
- 两者独立，满足任一即触发

---

### T3 · 定时兜底

**原理：** 没有触发信号时，用定时检查避免遗忘。

**规则：**
- 每 **2 周** 跑一次全量 skill 基线评估
- 时间点：cron 任务，周日 10:00

**触发后的动作：**
- 评估结果推送飞书，用户决定是否继续优化
